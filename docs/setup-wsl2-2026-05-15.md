# Jarvis 24/7 - WSL2 + Windows Task Scheduler

**Fecha:** 2026-05-15  
**Decision:** correr el core de Jarvis dentro de WSL2; Windows solo supervisa.  
**Costo:** USD 0. Requisito: el portatil debe estar encendido y la sesion de Windows iniciada.

## Arquitectura

```text
Windows host
  -> Task Scheduler
      -> scripts/jarvis-watchdog-wsl.ps1
          -> wsl -d Ubuntu -- node runtime/openclaw-jarvis/bin/jarvis-lifecycle.mjs status|start

WSL2 Ubuntu
  -> repo montado en /mnt/c/Users/Acer Nitro/Multi-LLM-Jarvis-Ecosistem
  -> .venv Linux
  -> uvicorn / FastAPI core
  -> Ollama + Whisper CPU/int8
```

Windows no ejecuta el core. Windows solo despierta el watchdog. El proceso real de Jarvis vive en Ubuntu/WSL2, con paths Linux, dependencias Linux y menos friccion con herramientas AI.

## Instalacion Base

1. Instalar WSL2 Ubuntu desde PowerShell:

```powershell
wsl --install -d Ubuntu
wsl --set-default-version 2
```

2. Abrir Ubuntu e instalar dependencias del sistema:

```bash
sudo apt update
sudo apt install -y python3 python3-venv python3-pip ffmpeg nodejs npm
```

3. Instalar Ollama siguiendo la guia oficial de `ollama.com/download/linux`. Despues:

```bash
ollama pull qwen2.5:3b
```

## Entorno Python Linux

Crear el venv en la raiz del monorepo, no dentro de `core/`, porque `jarvis-lifecycle.mjs` busca `.venv/bin/python` en `REPO_ROOT`.

```bash
cd "/mnt/c/Users/Acer Nitro/Multi-LLM-Jarvis-Ecosistem"
python3 -m venv .venv
.venv/bin/python -m pip install -U pip
.venv/bin/pip install -r core/requirements.txt
.venv/bin/python -m playwright install chromium
```

Notas:

- `core/requirements.txt` incluye `faster-whisper`, `yt-dlp`, `playwright` y `httpx`.
- `ffmpeg` se instala con `apt`, no con pip.
- Whisper debe funcionar aunque CUDA no este lista: `stt.py` intenta el dispositivo configurado y cae a CPU `int8`.

## Probar Lifecycle Dentro de WSL2

```bash
cd "/mnt/c/Users/Acer Nitro/Multi-LLM-Jarvis-Ecosistem"
node runtime/openclaw-jarvis/bin/jarvis-lifecycle.mjs status
node runtime/openclaw-jarvis/bin/jarvis-lifecycle.mjs start
node runtime/openclaw-jarvis/bin/jarvis-lifecycle.mjs status
node runtime/openclaw-jarvis/bin/jarvis-lifecycle.mjs stop
```

Logs:

```text
runtime/openclaw-jarvis/var/jarvis.log
runtime/openclaw-jarvis/var/jarvis.pid
```

`runtime/openclaw-jarvis/var/` esta ignorado por Git.

## Registrar Supervisor en Windows

Desde PowerShell:

```powershell
cd "C:\Users\Acer Nitro\Multi-LLM-Jarvis-Ecosistem\scripts"
powershell -ExecutionPolicy Bypass -File install-jarvis-supervisor.ps1 -Wsl -Distro Ubuntu
```

La tarea se llama `JarvisSupervisor` y corre al iniciar sesion + cada 5 minutos.

Logs del watchdog:

```text
C:\Users\Acer Nitro\Multi-LLM-Jarvis-Ecosistem\runtime\openclaw-jarvis\var\watchdog-wsl.log
```

Desinstalar:

```powershell
powershell -ExecutionPolicy Bypass -File install-jarvis-supervisor.ps1 -Uninstall
```

## Agent Browser de Vercel

Con Windows-native no era buena base para este proyecto por problemas de paths/sockets/daemon. Con WSL2 si es viable porque se ejecuta como Linux.

Uso recomendado:

1. Mantener Playwright como fallback actual del pipeline Reels.
2. Cuando WSL2 + supervisor esten estables, instalar Agent Browser dentro de Ubuntu.
3. Usarlo como etapa browser preferida antes de Playwright para bajar tokens y mejorar acciones por snapshot.

Referencia primaria: `https://github.com/vercel-labs/agent-browser`.

Comandos de referencia desde el README upstream:

```bash
cargo install agent-browser
agent-browser install
agent-browser open example.com
agent-browser snapshot
```

Seguridad:

- Usar perfil/sesion dedicada.
- No usar Chrome personal.
- No guardar cookies/tokens en el repo.

## Checklist Operativo

- `wsl -l -v` muestra Ubuntu en version 2.
- `node --version` funciona dentro de Ubuntu.
- `.venv/bin/python -c "import jarvis"` funciona desde el repo.
- `node runtime/openclaw-jarvis/bin/jarvis-lifecycle.mjs start` deja `/health` activo.
- Task Scheduler contiene `JarvisSupervisor`.
- `watchdog-wsl.log` muestra `ok`.
